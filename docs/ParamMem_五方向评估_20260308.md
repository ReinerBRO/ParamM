# ParamMem 五个方向评估（2026-03-08）

## 评估前提

- 当前代码基线仍是 `main_param.py` + `paramAgent.py` 两阶段框架：Phase1 生成/反思，Phase2 对失败题做反思增强与 memory 检索。
- 当前显式 memory 仍然很轻：`mem_bank.pkl` 仅维护 `positive_trajectories` 和 `negative_trajectories`，见 `paramAgent.py:150`。
- 当前 `ParamMem` 的论文主线仍然是“离线训练出的 parametric reflective prior，在推理时被读取”，而不是完整的 read-write memory system，见 `docs/papers/ParamMem.txt:410` 与 `docs/papers/ParamMem.txt:1050`。
- 当前 `Direction1` 类型工作更适合作为 `Phase2 execution/search kernel`，而不是整篇工作的主故事。

## 评估对象

本次评估的五个方向是：

1. `Counterfactual Parametric Memory`
2. `Hierarchical Parametric Memory`
3. `Dual-Use Parametric Memory`
4. `Continual Parametric Memory Consolidation`
5. `Parametric Memory as Latent Reasoning State`

## 总结结论

- 最适合做**主线系统故事**的方向，不是 `Direction1`，而是：
  - `Read-Write Closed-Loop ParamMem / MemoryOS`
- 在这五个新方向里，真正最值得保留的关系是：
  - `Continual Parametric Memory Consolidation` 不适合作为独立主线；它更适合作为 `Read-Write Closed-Loop ParamMem` 的技术核心。
- 如果 `Read-Write` 已经在另一个 worktree 里推进，那么**当前 worktree 最适合新开的方向**是：
  - `Dual-Use Parametric Memory`
- 第二优先的备选，是：
  - `Counterfactual Parametric Memory`

换句话说，这五个方向不应该平铺推进，而应该收敛成：

- `Read-Write Closed-Loop ParamMem / MemoryOS`：总主线
- `Continual Consolidation`：主线核心技术
- `Dual-Use Parametric Memory`：当前最适合独立开工的新方向
- `Counterfactual Parametric Memory`：第二优先的备选 worktree

## 1. Counterfactual Parametric Memory

### 核心想法

- 把 parametric memory 的学习目标，从静态 `problem -> reflective hint` 升级成动态的 `(problem, failed trajectory) -> better alternative branch`。
- memory 学的不只是“哪里可能错”，而是“如果当前这条推理分支失败，下一步更应该转向哪类 reasoning mode / repair mode”。

### 优势

- 最贴近 HumanEval / MATH 这类“失败后如何转向”是核心瓶颈的任务。
- 和当前 Phase2 非常兼容：现有 Phase2 已经有失败轨迹、反馈、反思、重试，只是还没有把“更优替代分支”显式学成 parametric memory。
- 比单纯 pitfall 更强，因为它学的是“转向策略”，不是“静态提醒”。

### 问题

- 训练信号更难构造。你需要高质量的 `(failed trajectory -> better branch)` 样本，而不是普通 reflective text。
- 如果没有足够强的写回与筛选机制，这条线很容易退化成“失败样本上的 prompt 改写”，失去 parametric memory 的主角地位。

### 结论

- 值得做。
- 但如果 `Read-Write` 主线已经单独开工，这条线更适合排在 `Dual-Use Parametric Memory` 之后。

## 2. Hierarchical Parametric Memory

### 核心想法

- 不再让单一 ParamMem 同时承载所有尺度的 reasoning knowledge，而是拆成：
  - `Concept Memory`
  - `Strategy Memory`
  - `Pitfall / Error Memory`
- 让 parametric memory 以分层方式作用于推理，而不是只生成单层 reflective text。

### 优势

- 学术故事很强，系统味很重。
- 非常适合写成“现有 ParamMem 把不同粒度的 reasoning knowledge 混在一个 module 里，表达能力受限”。
- 天然兼容 code / math / QA 三类任务。

### 问题

- 当前代码和数据都不具备这样的层级 supervision。
- 现有轨迹记录更接近单层文本反思与成功/失败样本，不足以直接支撑三层 parametric memory 的训练。
- 工程跨度大，短期最难落地。

### 结论

- 论文故事很好，但当前最不现实。
- 不建议当前 worktree 优先做。

---

## 3. Dual-Use Parametric Memory

### 核心想法

- 同一个 parametric memory 不只服务于生成，还服务于自我批判。
- 即：
  - `memory_for_generation`
  - `memory_for_critique`
- 让 parametric memory 同时给 solver 和 critic 提供长期知识，而不是只做单向提示。

### 优势

- 和当前代码最兼容。
- 当前系统已经有自然的 `generate -> evaluate -> reflect` 闭环，只是 reflective prior 主要注入在生成端。把 parametric memory 也用于 critique，是很自然的延展。
- 对 code、math、QA 都有合理落点：
  - code：bug localization / likely failure dimension
  - math：invalid step detection
  - QA：unsupported answer / weak evidence chain detection

### 问题

- 需要把“critic memory”训练目标定义清楚，避免退化成普通 reward model 或 verifier 提示器。
- 必须明确它仍然是 parametric memory，而不是独立 critic model。

### 结论

- 如果 `Read-Write` 已经在另一个 worktree 推进，**这是当前最适合新开 worktree 的方向**。
- 原因是：
  - 兼容性最好；
  - 工程跨度适中；
  - 故事性足够强；
  - 很可能带来真实涨点，尤其在 QA 和 code critique 上。

---

## 4. Continual Parametric Memory Consolidation

### 核心想法

- 研究重点不再是“能不能写回”，而是“如何稳定地把在线经验巩固成 parametric memory”。
- 真正的关键步骤是：
  - `promotion`
  - `compression`
  - `consolidation`
  - `retention`

### 优势

- 非常直接地回应 reviewer 最常见的质疑：
  - “这不就是 online finetune 吗？”
- 和 `ParamMem` 论文里 self-teaching、sample efficiency 的支点最一致。
- 一旦做得好，会同时提升 Phase1 与 Phase2，而不只是继续压榨 Phase2。

### 问题

- 它其实不是最适合单独拉成一条平行主线。
- 从系统逻辑上看，它更像 `Read-Write Closed-Loop ParamMem / MemoryOS` 的技术核心，而不是另一套独立故事。

### 结论

- 不建议把它单独当成“当前 worktree 的独立方向”。
- 更合理的定位是：
  - `Read-Write Closed-Loop ParamMem / MemoryOS` 的核心内核。

---

## 5. Parametric Memory as Latent Reasoning State

### 核心想法

- 不再让 ParamMem 直接输出 textual reflective prior，而是让它先预测 latent reasoning state：
  - 当前处于哪种 reasoning mode
  - 离成功差什么
  - 当前失败属于哪类 latent state
- solver 再在该 latent state 条件下继续推理。

### 优势

- 跨任务统一性最强。
- 论文味最浓，很容易写成“从 textual reflection memory 升级到 state-aware parametric reasoning system”。

### 问题

- 当前最不现实。
- 现有系统是文本 reflection 驱动，不是 latent-state 驱动。
- 你现在没有 state supervision，也没有合适的 latent controller 或 belief update 基础设施。

### 结论

- 是非常好的长期方向。
- 但当前不适合作为新 worktree 立即开工。

## 最终优先级

### 研究主线优先级

1. `Read-Write Closed-Loop ParamMem / MemoryOS`
2. `Dual-Use Parametric Memory`
3. `Counterfactual Parametric Memory`
4. `Hierarchical Parametric Memory`
5. `Parametric Memory as Latent Reasoning State`

### 技术定位修正

- `Continual Parametric Memory Consolidation` 不再作为独立主线排序。
- 它应被并入：
  - `Read-Write Closed-Loop ParamMem / MemoryOS`
  - 作为其最核心的技术机制。

### 当前 worktree 建议

在 `Read-Write` 已经由另一个 worktree 推进的前提下：

- 当前 worktree 最推荐方向：`Dual-Use Parametric Memory`
- 当前 worktree 第二推荐方向：`Counterfactual Parametric Memory`

## 一句话结论

- `Direction1` 应继续降级为 `execution/search kernel`。
- `Read-Write Closed-Loop ParamMem / MemoryOS` 负责总故事。
- 如果当前 worktree 还要再开一条真正值得做的新方向，最应该做的是：
  - `Dual-Use Parametric Memory`
