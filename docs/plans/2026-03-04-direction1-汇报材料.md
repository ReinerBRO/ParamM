## F. 3分钟口播稿（可直接读）

我们这次做的不是简单多跑几轮，而是把 ParamAgent 的 Phase2 改成了验证驱动搜索。具体是：在 `phase2_search_mode=search` 下，使用 SearchEngine 扩展候选，并按“通过概率 + 多样性 - 复杂度惩罚”排序，同时结合可见测试、生成测试和静态检查来做验证。
结果上，Direction1 在全量数据、24 worker 下，HumanEval 达到 `146/164 (89.02%)`，MBPP 达到 `309/397 (77.83%)`。相比 ParamAgent-plus 的 `133/164` 和 `299/397`，两个数据集都提升，其中 MBPP 超过我们验收线 `299/397`。
所以结论是：Direction1 的创新是有效的，并且与论文强调的“反思多样性驱动收益”一致。

## J. 原版 ParamAgent 源码：Phase2 具体流程（root `paramAgent.py`）

1. **进入 second pass 并做快照**：加载 `memory_bank/logs`，写 `first_stage_json`；若 `phase1_only=True` 则直接返回。
   - 证据：`paramAgent.py:548-559`
2. **筛选 Phase2 输入**：只保留 `is_solved=False` 且 `stage2=False` 的第一阶段失败题。
   - 证据：`paramAgent.py:562-566`
3. **加载全局正轨迹（可选）**：从 `global_mem_bank_path` 读取 `positive_trajectories`。
   - 证据：`paramAgent.py:568-580`
4. **逐题修复主循环**：对失败题按 `pass_at_k` 循环，准备 prompt/tests。
   - 证据：`paramAgent.py:588-637`
5. **首次检索与生成（k=1）**：
   - 检索池：`global_positive_trajectories + memory_bank["positive_trajectories"]`
   - prompt 相似度检索 `k=1`，构造增强 prompt 后生成初始实现。
   - 证据：`paramAgent.py:640-646, 652-655, 673-680`
6. **两级验证**：先执行可见/合成测试，再过隐藏真实测试；通过即写入正轨迹并结束该题。
   - 证据：`paramAgent.py:689-714`
7. **未通过则进入线性 reflexion 迭代**：`while cur_iter < max_iters`。
   - 证据：`paramAgent.py:716-718`
8. **每轮最多尝试 2 条 reflection + 检索（k=1）**：
   - 优先按 `reflection_embedding` 检索 `k=1`；
   - 无可用 reflection 向量时回退到 prompt 检索 `k=1`。
   - 证据：`paramAgent.py:747-780`
9. **基于 reflection 生成新实现并打分**。
   - 证据：`paramAgent.py:799-838`
10. **写入 memory 与线性推进**：
    - 通过：写 `positive_trajectories`；
    - 未通过：写 `negative_trajectories`；
    - 若继续迭代，从候选按分数加权采样 1 个作为下一轮当前解（单轨迹线性推进）。
    - 证据：`paramAgent.py:850-910`
11. **落盘与日志更新**：回写 `stage2` 标记、成本/token、轨迹信息，写 `second_stage_json` 与 `mem_bank.pkl`。
    - 证据：`paramAgent.py:924-962`

**一句话结论**：原版 Phase2 是“失败题逐题修复 + 检索增强 + 反思迭代 + 单轨迹线性推进”，不是 beam/tree search。
