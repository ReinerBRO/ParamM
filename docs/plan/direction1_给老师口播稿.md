## 一、ParamAgent 的定位与原始问题

ParamAgent 的定位，不是通过“换更大模型”来提分，而是研究：
在同一基础模型条件下，如何提升 Agent 的自我反思与自我修复能力。

它要解决的痛点主要有三个：
1) 传统 Reflexion 流程主要依赖单题内反思，跨题经验复用弱；
2) 反思容易同质化，出现重复试错；
3) 代码任务中线性重试成本高，但有效修复路径命中率不稳定。

## 二、ParamAgent 如何解决

ParamAgent 采用两阶段框架：

- Phase1（ParamAgent）：
  对每道题执行“生成—验证—反思”迭代，
  并把成功轨迹和失败轨迹沉淀到 memory bank。

- Phase2（ParamAgent-plus）：
  只针对 Phase1 失败题再修复，
  引入 cross-sample memory，从相似历史轨迹检索可迁移经验，
  提升二次修复效果。

在编程任务上，中间迭代使用可见/合成测试反馈，
最终用隐藏测试计算 Pass@1。

所以，这个工作的本质是：
把“只在当前样本内反思”，升级为“可积累、可迁移的反思记忆系统”。

## 三、我们的进一步工作：Direction1

我们观察到：
原框架在 Phase2 虽然已有 cross-sample memory，
但候选修复过程仍偏线性，缺少系统性的“验证驱动选择”。

因此我们做了 Direction1：
把 Phase2 从线性迭代，升级为“验证驱动搜索式修复”。

核心改动有四点：

1) 在 Phase2 引入 search mode；
2) 用 SearchEngine 管理候选扩展与选择（beam、深度、节点预算、early-stop）；
3) 设计组合评分：
   候选分数 = 通过概率 + 多样性收益 - 复杂度惩罚；
4) 验证器融合三类信号：可见测试、生成测试、静态检查，
   并用哈希缓存减少重复验证开销。

同时保留稳定性机制：
当搜索分支异常时，自动回退到线性流程，避免任务中断。

## 四、实验结果（全量数据，24 worker）

Direction1（Phase2）结果：
- HumanEval：146/164，89.02%
- MBPP：309/397，77.83%

与 ParamAgent-plus（Phase2）对比：
- HumanEval：133/164 → 146/164（+7.92 个百分点）
- MBPP：299/397 → 309/397（+2.52 个百分点）

结论是：
Direction1 在两个代码基准上都实现了稳定提升，
并且 MBPP 超过了我们设定的验收线 299/397。

## 五、总结与下一步

总结一句话：
Direction1 在保留 ParamAgent 记忆机制优势的基础上，
通过“验证驱动搜索”进一步提升了 Phase2 修复质量。
